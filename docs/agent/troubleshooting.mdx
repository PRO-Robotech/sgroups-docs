# Отладка несоответствий nft ↔ API

При расследовании «почему правило не работает» удобно идти по тем же доменам,
по которым устроен тестовый набор `sgroups-test-go`: **api → entities → chains → routing →
unirule (placement / matchers / modifiers) → ordering**. Если сломан верхний уровень,
нижние почти всегда сломаются тоже.

## Шаг 1. API — сервер принял правило?

```bash
curl -X POST http://localhost:9006/v1/rules/list \
  -H 'Content-Type: application/json' \
  -d '{"selectors":[{"fieldSelector":{"name":"my-rule","namespace":"production"}}]}'
```

Если правила нет в ответе или статус апсерта был не `accepted` — расхождение начинается на
сервере; дальше идти бессмысленно.

Типичные причины:
- недопустимая комбинация `endpoints.*.type` и `traffic` (см. матрицу поддержанных пар в [«Соответствие правил nft»](./rule-mapping));
- `transport` обязателен для текущей пары типов, но не передан;
- `transport` передан там, где не должен (например, `SERVICE → SERVICE` — транспорт наследуется от local-сервиса);
- `BOTH` для `AG → ICMP` (запрещено).

## Шаг 2. Entities — наборы существуют и наполнены?

Имя таблицы агента — динамическое (`main-${timestamp}`), поэтому удобнее искать ее
через `nft list ruleset` и фильтровать `grep`, либо сначала вычислить имя:

```bash
# найти имя текущей таблицы агента
TABLE=$(sudo nft list tables | awk '/inet main-/{print $3; exit}')

sudo nft list set inet "$TABLE" NetIPv4-addressGroup-production/web-servers
```

Что искать:
- **набор отсутствует** — AG не привязан к этому узлу через `HostBinding`;
- **набор пустой** (`elements = { }`) — ни один привязанный `Host` не имеет IPv4 в `spec.IPs.IPv4`
  (нужен `POST /v1/hosts/upd-ips`);
- **набор наполнен, но не теми IP** — `HostBinding` указывает на другой `Host` или у хоста
  изменились IP, а агент еще не успел синхронизироваться (см. `sync.interval`).

Для FQDN-наборов проверяйте разрешение DNS:

```bash
sudo nft list set inet "$TABLE" NetIPv4-fqdn-example.com
nslookup example.com  # сравните с содержимым набора
```

## Шаг 3. Chains — цепочки созданы?

```bash
sudo nft list chains inet "$TABLE"
```

Должны быть:
- `INGRESS-MAIN` и `EGRESS-MAIN` — всегда;
- по паре `INGRESS-ag-<ns>/<name>` / `EGRESS-ag-<ns>/<name>` на каждый локальный AG.

Если для какого-то локального AG цепочки нет, агент не считает его локальным. Проверьте,
что для этого AG существует `HostBinding` на узел, на котором работает агент.

## Шаг 4. Routing — есть jump в per-AG цепочку?

```bash
sudo nft list chain inet "$TABLE" INGRESS-MAIN
sudo nft list chain inet "$TABLE" EGRESS-MAIN
```

Тело должно содержать:

```nft
ip saddr @NetIPv4-addressGroup-<ns>/<name> jump INGRESS-ag-<ns>/<name>
ip daddr @NetIPv4-addressGroup-<ns>/<name> jump EGRESS-ag-<ns>/<name>
```

Если такой записи нет, это типичный случай «правила есть, но не работают». Обычно помогает
перезапуск агента.

## Шаг 5. Unirule placement — правило в **той** цепочке?

Найти строку:

```bash
sudo nft list ruleset | grep -E "(saddr|daddr).*(@NetIPv4|203\.0\.113|fqdn-example)"
```

Сверьте с [правилами выбора цепочки](./rule-mapping#общая-форма-строки):
- `local.type=AG`, `traffic=INGRESS` — `INGRESS-ag-<local.ns>/<local.name>`;
- `local.type=SERVICE` — per-AG того AG, к которому привязан local-сервис.

Частые ошибки:
- правило оказалось в **обеих** цепочках (`INGRESS` и `EGRESS`) при `traffic=INGRESS` —
  это `unirule-placement: unexpected-chain`;
- правило продублировалось в одной цепочке — `duplicate-in-expected-chain`,
  обычно из-за повторного апсерта без удаления старого правила.

## Шаг 6. Matchers — правильные saddr/daddr/dport/icmp type?

```bash
sudo nft list chain inet "$TABLE" INGRESS-ag-production/web-servers
```

Сверьте по таблице из [«Соответствие правил nft»](./rule-mapping):
- `saddr` / `daddr` ссылается на правильный набор;
- `protocol` совпадает с `transport.protocol`;
- `dport` / `icmp type` совпадают с `transport.entries[].ports` / `.types`.

## Шаг 7. Modifiers — action / trace / logs

В строке должны быть:
- `accept` для `ALLOW`, `drop` для `DENY`;
- `meta nftrace set 1`, если `linux-agent.sgroups.io/trace = "true"`;
- `log level debug`, если `linux-agent.sgroups.io/logs = "true"`.

См. [«Аннотации»](./annotations).

## Шаг 8. Ordering — порядок строк по priority

Внутри per-AG цепочки строки идут **в порядке возрастания** аннотации
`linux-agent.sgroups.io/priority`. Проверить можно так:

```bash
sudo nft -a list chain inet "$TABLE" EGRESS-ag-production/web-servers
```

Аргумент `-a` выводит идентификаторы (handles) строк — порядок в выводе совпадает с порядком исполнения.

Если порядок нарушен, это **баг агента**: отчет `implementation-audit.json`
(категория `priority_inversions`) ловит такие случаи в тестовой матрице.

## Полная карта проверок

Из тестового набора `sgroups-test-go`:

| Domain | Что проверяет | Как воспроизвести руками |
|---|---|---|
| `api` | каждый кейс получил ожидаемый HTTP-статус от `/v1/rules/upsert` | `curl POST` и проверка кода ответа |
| `entities` | для каждого AG/SVC из setup есть набор nft | `nft list sets inet "$TABLE"` |
| `chains` | `INGRESS-MAIN`, `EGRESS-MAIN`, `INGRESS-ag-*`, `EGRESS-ag-*` существуют | `nft list chains inet "$TABLE"` |
| `routing` | в MAIN-цепочке есть `jump` в каждую локальную AG-цепочку | `nft list chain inet "$TABLE" INGRESS-MAIN` |
| `unirule-placement` | правило находится в **нужной** цепочке и **только** в ней | `grep` по выводу `nft list ruleset` |
| `unirule-matchers` | условия совпадения (`saddr`/`daddr` + `port`/`type`) присутствуют в nft | то же |
| `unirule-modifiers` | `action` / `trace` / `logs` совпадают с аннотациями | сравнение строки и аннотаций |
| `ordering` | `priority` возрастает в порядке размещения в цепочке | `nft -a list chain …` |
| `known-gaps` | задокументированные расхождения | — |

Зеленый прогон означает, что во всех восьми доменах 0 ошибок. Красный — есть реальный баг
сервера или несоответствие данных.

## Запуск автоматической проверки

Если узел доступен через `kubectl exec`, запустите тестовый набор:

```bash
git clone https://github.com/PRO-Robotech/sgroups-test-go
cd sgroups-test-go
make build
./bin/sgnft generate --out resources.json
./bin/sgnft apply  --resources resources.json --report-dir reports
sleep 600
./bin/sgnft verify --resources resources.json --report-dir reports \
  --api-results reports/api-results.json
```

Артефакты после verify:

| Файл | Зачем смотреть |
|---|---|
| `reports/error-report.md` | сводка по доменам и разбивка по `(family, traffic, transport)` |
| `reports/case-error-report.md` | подробный список ошибок по конкретным кейсам с диагностикой |
| `reports/bug-report.md` | сгруппированные fingerprints с каноническим воспроизведением |
| `reports/domain-audit.json` | результаты девяти доменов в JSON |
| `reports/implementation-audit.json` | строгие находки (`missing` / `duplicates` / `wrong-action` / `trace-mismatch` / `priority-inversion`) |

См. [README sgroups-test-go](https://github.com/PRO-Robotech/sgroups-test-go) для деталей.

## Логи агента

```bash
# systemd
journalctl -u sg-agent -f

# docker
docker logs -f sg-agent

# kubernetes
kubectl logs -n sgroups-system -l app=sgroups-agent -f --tail 100
```

Ключевые события:

| Уровень | Сообщение | Значение |
|---|---|---|
| `info` | `sync completed` | успешный цикл |
| `warn` | `dns resolve failed` | FQDN не разрешился — соответствующий набор не пополнится |
| `error` | `nft apply failed` | транзакция отменена; в логе должен быть стектрейс nft |
| `error` | `server connection lost` | gRPC-соединение с сервером разорвано, агент ждет по экспоненциальной паузе |
