# Структура nftables, которую строит агент

Все объекты находятся в одной таблице семейства `inet` с динамическим именем
`main-${ts:1714567890}`. Внутри нее агент создает **базовые цепочки**, **per-AG
цепочки** и **наборы (sets)** с IP-данными. Имена цепочек и наборов строго детерминированы —
именно по ним инструменты проверки сопоставляют `UniRule` с конкретной строкой nft.

Все примеры — **шаблоны**: запись `${имя:образец}` означает плейсхолдер с именем `имя`
и примером значения `образец`.

Префиксы `l-` / `r-` обозначают **local** / **remote** (стороны правила), а не
left/right (позиции в nft-выражении).

- `${ts:1714567890}` — timestamp таблицы
- `${l-ns:prod}` — namespace локальной AG
- `${l-ag:web}` — имя локальной AG
- `${r-ns:partner}` — namespace удаленной AG
- `${r-ag:billing}` — имя удаленной AG
- `${svc:api}` — имя сервиса
- `${fqdn:example.com}` — FQDN
- `${cidr:10.0.0.0/24}` — CIDR
- `${ip:10.0.0.1}` — IP хоста
- `${port:8080}` — порт

## Таблица

```nft
table inet main-${ts:1714567890} { … }
```

- семейство `inet` — единое для IPv4/IPv6 (фактически правила работают с IPv4);
- имя таблицы — `main-${ts:1714567890}`. Меняется на каждом новом цикле сборки и
  **не настраивается** через конфигурацию;
- свои таблицы агент опознает по префиксу `main-`. Сторонние таблицы nftables агент не
  трогает.

## Базовые цепочки

| Цепочка | Hook | Priority | Policy | Назначение |
|---|---|---|---|---|
| `INGRESS-MAIN` | `prerouting` | `-130` | `drop` | принимает входящий трафик и делает `jump` в per-AG цепочку, если `saddr` принадлежит локальному AG |
| `EGRESS-MAIN` | `postrouting` | `300` (`conntrack helper`) | `drop` | обрабатывает исходящий трафик и делает `jump` в per-AG цепочку, если `daddr` принадлежит локальному AG |

Policy цепочек:

- по умолчанию — `drop` (все, что не подошло ни под одно правило, отбрасывается);
- если в конфигурации агента включен `def-policy-accept: true` либо все локальные AG имеют
  `defaultAction = ALLOW`, policy переключается на `accept`.

Первой строкой в каждой MAIN-цепочке агент добавляет
`ct state established,related accept` — это пропускает уже установленные сессии
мимо последующих правил.

Дальше тело базовых цепочек состоит только из инструкций `jump`, по одной на каждый
локальный AG, привязанный к этому узлу через `HostBinding`:

```nft
chain INGRESS-MAIN {
    type filter hook prerouting priority -130; policy drop;
    ct state established,related accept
    ip saddr @NetIPv4-addressGroup-${l-ns:prod}/${l-ag:web} jump INGRESS-ag-${l-ns:prod}/${l-ag:web}
}
chain EGRESS-MAIN {
    type filter hook postrouting priority 300; policy drop;
    ct state established,related accept
    ip daddr @NetIPv4-addressGroup-${l-ns:prod}/${l-ag:web} jump EGRESS-ag-${l-ns:prod}/${l-ag:web}
}
```

:::tip Что проверяет verify
Тестовый набор `sgroups-test-go` (домен `routing`) для каждого локального AG проверяет
наличие `jump INGRESS-ag-${l-ns:prod}/${l-ag:web}` в `INGRESS-MAIN` и аналогично для
`EGRESS-MAIN`. Если такого `jump` нет — правила в per-AG цепочке существуют, но трафик до
них не доходит.
:::

## Per-AG цепочки

На каждый AG, привязанный к узлу, агент создает пару цепочек:

```nft
chain INGRESS-ag-${l-ns:prod}/${l-ag:web} { … }
chain EGRESS-ag-${l-ns:prod}/${l-ag:web} { … }
```

Имя цепочки **включает namespace и имя AG со слэшем**. Это важно: инструмент проверки
ищет правила именно по такому ключу.

Внутрь цепочки агент пишет строки nft — по одной на каждый
порт / тип / CIDR / FQDN из `UniRule`. Структура строки зависит от направления:
для INGRESS используется `saddr` (откуда пришел пакет), для EGRESS — `daddr`
(куда уходит пакет).

```nft
chain INGRESS-ag-${l-ns:prod}/${l-ag:web} {
    ip [saddr <left>] <protocol> <port-or-type> [meta nftrace set 1] [log level debug] <accept|drop>
}

chain EGRESS-ag-${l-ns:prod}/${l-ag:web} {
    ip [daddr <right>] <protocol> <port-or-type> [meta nftrace set 1] [log level debug] <accept|drop>
}
```

Конкретные виды `<left>` / `<right>` для каждого типа `UniRule` — в разделе
[«Соответствие правил nft»](./rule-mapping).

## Наборы (sets)

Наборы — это таблицы IP-адресов, которые наполняются IP-адресами хостов и подсетей. Агент создает их
с типом `ipv4_addr` и флагом `interval` (чтобы можно было хранить и одиночные адреса, и CIDR).

| Имя набора | Источник | Как наполняется |
|---|---|---|
| `NetIPv4-addressGroup-${l-ns:prod}/${l-ag:web}` | `AddressGroup` | объединение IPv4 всех `Host`, привязанных через `HostBinding` к этому AG, плюс CIDR из `Network`, привязанных через `NetworkBinding` |
| `NetIPv4-svc-${l-ns:prod}/${svc:api}` | `Service` | объединение IPv4 всех `Host`, привязанных к AG, к которому привязан этот сервис через `ServiceBinding` |
| `NetIPv4-fqdn-${fqdn:example.com}` | inline в `UniRule` (`remote.type = FQDN`) | результат A-записи DNS, кэшируется на `dns.cache-ttl` |

```nft
set NetIPv4-addressGroup-${l-ns:prod}/${l-ag:web} {
    type ipv4_addr
    flags interval
    elements = { ${ip:10.0.0.1} }
}
```

CIDR-значения (`remote.type = CIDR`) подставляются в строку правила **inline**, без отдельного набора:

```nft
chain EGRESS-ag-${l-ns:prod}/${l-ag:web} {
    ip daddr { ${cidr:10.0.0.0/24} } tcp dport ${port:8080} accept
}
```

## Полный пример

Минимальный набор правил на одном AG с одним правилом `tcp dport ${port:8080} accept`:

```nft
table inet main-${ts:1714567890} {
    set NetIPv4-addressGroup-${l-ns:prod}/${l-ag:web} {
        type ipv4_addr
        flags interval
        elements = { ${ip:10.0.0.1} }
    }
    chain INGRESS-MAIN {
        type filter hook prerouting priority -130; policy drop;
        ct state established,related accept
        ip saddr @NetIPv4-addressGroup-${l-ns:prod}/${l-ag:web} jump INGRESS-ag-${l-ns:prod}/${l-ag:web}
    }
    chain EGRESS-MAIN {
        type filter hook postrouting priority 300; policy drop;
        ct state established,related accept
        ip daddr @NetIPv4-addressGroup-${l-ns:prod}/${l-ag:web} jump EGRESS-ag-${l-ns:prod}/${l-ag:web}
    }
    chain INGRESS-ag-${l-ns:prod}/${l-ag:web} {
        ip saddr @NetIPv4-addressGroup-${l-ns:prod}/${l-ag:web} tcp dport ${port:8080} accept
    }
    chain EGRESS-ag-${l-ns:prod}/${l-ag:web} {
    }
}
```

## Соответствие имен

Соответствие API-объектов именам nft (плейсхолдеры — это значения, которые
подставляются агентом из API):

| Объект API | Имя в nft |
|---|---|
| `AddressGroup{namespace, name}` | набор `NetIPv4-addressGroup-${l-ns:prod}/${l-ag:web}`, цепочки `INGRESS-ag-${l-ns:prod}/${l-ag:web}` / `EGRESS-ag-${l-ns:prod}/${l-ag:web}` |
| `Service{namespace, name}` | набор `NetIPv4-svc-${l-ns:prod}/${svc:api}` |
| `UniRule{remote.type=FQDN, value}` | набор `NetIPv4-fqdn-${fqdn:example.com}` |
| `UniRule{remote.type=CIDR, value}` | inline в строке (`ip saddr { ${cidr:10.0.0.0/24} }` либо `ip daddr { ${cidr:10.0.0.0/24} }`) |

:::warning Если набор пуст
Если ни один `Host` не привязан к AG (нет `HostBinding`), набор создается пустым:
правила есть, но совпадений по адресу не будет. Домен проверки `entities`
сообщает о таких случаях как `missing`.
:::
