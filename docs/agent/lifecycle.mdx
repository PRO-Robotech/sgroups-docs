# Жизненный цикл агента

Страница описывает, что именно делает агент с момента запуска и как реагирует на
события — изменения в базе сервера, изменения локальной сети, истечение времени
жизни DNS-записей. Описание полностью прикладное: только что и в каком порядке
происходит, без отсылок к конкретным реализациям.

## Запуск агента: что происходит по шагам

1. **Чтение конфигурации.** Агент собирает параметры из трёх источников по приоритету:
   аргументы командной строки → переменные окружения с префиксом `NFT_` → файл
   конфигурации. На этом шаге становятся известны: адрес сервера, идентификатор
   локального хоста, DNS-серверы, имя сетевого namespace (если используется),
   стратегия резолва FQDN, политика по умолчанию (`accept` / `drop`), интервалы.

2. **Поднятие телеметрии.** Запускается локальный HTTP-сервер с метриками Prometheus,
   healthcheck-эндпоинтом и pprof-профилировщиком. С этого момента состояние агента
   видимо снаружи.

3. **Подписка на события сетевого стека.** Агент открывает netlink-канал и снимает
   текущее состояние интерфейсов и адресов локального узла. Все будущие изменения
   (поднялся / упал интерфейс, добавился / удалился IP) будут приходить ему
   автоматически.

4. **Подключение к серверу.** Открывается gRPC-канал до `sg-server`. Сразу же
   запускается стрим подписки на статус: сервер сам будет присылать сигналы
   "что-то изменилось в БД".

5. **Поиск себя на сервере.** Агент берёт идентификатор локального хоста из
   конфигурации и запрашивает у сервера ресурс `Host` с таким именем.
   - Если хост найден — берутся его атрибуты и список привязанных групп адресов.
   - Если хост **не найден** — это нештатно для production, но не фатально.
     Агент пишет предупреждение и продолжает работать; ruleset на узле останется
     минимальным, потому что нет точки входа в граф ресурсов.

6. **Публикация информации о себе.** Агент отправляет на сервер актуальный
   список IPv4/IPv6 локальных интерфейсов, имя ОС, hostname и прочую сводку —
   чтобы серверу было что показать в API статусов. После каждой успешной
   пересборки правил эта информация обновляется заново.

7. **Готовность к работе.** Агент ждёт двух сигналов: "сетевое состояние получено"
   и "сервер сообщил версию своих данных". Как только оба собраны — запускается
   первый цикл сборки правил.

## Сборка ruleset: обход графа ресурсов

Имея ресурс `Host`, агент рекурсивно вытягивает всё, что нужно для построения
firewall'а на этом узле. Порядок именно такой:

1. **Локальный `Host` → группы адресов.** В привязках хоста (`HostBinding`) указан
   список локальных AG — это те AG, которые "живут" на этом узле. Все они
   тянутся целиком.

2. **Локальные AG → соседние хосты.** Каждый локальный AG может ссылаться на
   набор `HostBinding`'ов других хостов. Их IP должны попасть в nft-set этого AG —
   агент идёт за их данными.

3. **Локальные AG → сети.** Если к AG привязаны `Network`-объекты (через
   `NetworkBinding`), их CIDR'ы тоже попадают в тот же nft-set.

4. **Локальные AG → сервисы.** Каждый AG может иметь привязки `ServiceBinding` —
   агент тянет соответствующие сервисы целиком (с их транспортами).

5. **Локальные AG и сервисы → правила.** На этом шаге собираются `UniRule`,
   которые ссылаются на любые из найденных AG/сервисов — как на локальную, так и
   на удалённую сторону.

6. **Удалённая сторона правил → донасос данных.** Если в правиле есть
   `remote.type = ADDRESS_GROUP` или `remote.type = SERVICE`, и эти AG/сервис
   **не локальны** для этого узла, агент догружает их содержимое
   (привязанные хосты, сети, IP) — иначе nft-set удалённой стороны окажется
   пустым и трафик не будет матчиться.

7. **Резолв FQDN.** Из правил выбираются все `remote.type = FQDN`, для них
   запрашиваются A/AAAA-записи через DNS из конфигурации. Время жизни записи
   запоминается — позже агент перерезолвит её ровно к моменту истечения.

8. **Снимок данных.** Все собранные ресурсы складываются в одну
   in-memory структуру — "снимок". Этот снимок сравнивается с тем, что было
   применено в прошлый раз. Если они одинаковы — пересборка nft-таблицы
   пропускается полностью (см. ниже).

9. **Сборка nft-таблицы.** Из снимка строится новая таблица семейства `inet` с
   динамическим именем `main-${timestamp}`: набор set'ов, базовые цепочки
   `INGRESS-MAIN` / `EGRESS-MAIN`, per-AG цепочки и правила в них (см.
   [«Структура nftables»](./nft-layout) и [«Соответствие правил nft»](./rule-mapping)).

## Защита от шквала событий

Перед тем как агент запустит цикл сборки, он даёт **окну изменений утихнуть**.
В системе два таких окна — для двух разных источников событий.

### Окно ожидания изменений с сервера

Параметр конфигурации:

```yaml
extapi:
  svc:
    sgroups:
      sync-status:
        interval: 10s   # default
```

Сервер отдаёт сигналы об изменениях через gRPC-стрим — в момент любой записи в БД.
При массовых правках это могут быть десятки уведомлений в секунду. Агент **не
обрабатывает каждое**. Он держит таймер с шагом `interval`: все сигналы, попавшие в
текущее окно, сворачиваются в один — в конце окна срабатывает ровно одна
обработка.

Поведение по сценариям:

- Один редкий сигнал — обработка примерно через `interval/2` после события.
- Шквал из 100 сигналов за 8 секунд — будет одна обработка по истечении
  10-секундного окна, без пятидесяти промежуточных пересборок.
- Уменьшать `interval` стоит только если важна минимальная задержка применения и
  допустима повышенная нагрузка от пересборок. Увеличивать — для очень нагруженных
  установок, где правки идут постоянным фоном.

### Окно ожидания изменений в сети узла

Параметр конфигурации:

```yaml
netlink:
  watcher:
    linger: 10s   # default
```

Когда Linux переконфигурирует интерфейсы (например, при поднятии нового пода или
переключении сетевых namespace), netlink выдаёт серию сообщений за миллисекунды.
Параметр `linger` — это длительность окна, в течение которого агент копит
netlink-уведомления, прежде чем отдать их в обработку одним пакетом.

Поведение аналогичное: при шквале изменений ruleset перестраивается один раз в
конце окна, а не на каждый отдельный пакет от ядра.

### Полный цикл с учётом окон

1. **Что-то меняется** — у сервера в БД или в сетевом стеке узла.
2. **Окно копит сигналы** в течение `interval` / `linger` секунд.
3. **По истечении окна** агент скачивает полный снимок ресурсов этого узла.
4. **Сравнивает со снимком прошлого цикла**. Сравнение глубокое — поэлементно по
   всем полям.
5. **Если равно** — пишет в лог "состояние не изменилось", ruleset не трогает,
   ждёт следующего сигнала.
6. **Если не равно** — собирает новую nft-таблицу.

Сравнение снимков работает поверх окон и страхует от лишних пересборок даже при
изменениях, которые в окно попали:

- Кто-то поменял правило, не касающееся этого узла → сигнал прилетит, снимок
  придёт, но окажется идентичен предыдущему — пересборки не будет.
- Linux добавил адрес на интерфейсе, не относящемся к локальному `Host` → событие
  попадёт в окно netlink, но содержимое снимка хоста не изменится — пересборки
  не будет.

## Атомарное применение правил

Самый чувствительный момент — **переключение** старого ruleset'а на новый. Агент
гарантирует, что в течение всего этого процесса трафик не "проваливается":
старая таблица обслуживает пакеты до тех пор, пока новая не готова целиком.

Алгоритм переключения по шагам:

1. **Создаётся новая таблица** с уникальным именем (содержит timestamp), сразу же
   помеченная как "спящая". Ядро её **не использует** для фильтрации — она
   видна в `nft list ruleset`, но не задействована.
2. **Заполнение спящей таблицы.** В неё постепенно добавляются: set'ы (адреса AG,
   сервисов, FQDN, локального хоста), базовые цепочки `INGRESS-MAIN` /
   `EGRESS-MAIN`, per-AG цепочки, правила. Каждый шаг — небольшая
   netlink-транзакция; при ошибке он повторяется с экспоненциальным backoff.
3. **Активация.** Один атомарный шаг снимает с новой таблицы статус "спящая" —
   ядро видит её как полноценный фильтр.
4. **Удаление старых.** Ровно следом удаляются все прежние таблицы агента.
   Сторонние nftables-конфиги (созданные другими сервисами) не трогаются — у
   таблиц агента специфичный префикс имени.

Что произойдёт, если на каком-то шаге случится сбой:

- Сбой **до активации** — спящая таблица остаётся незавершённой. Агент удаляет
  её и работает дальше со старой. Видимый эффект — нулевой, никакого
  даунтайма.
- Сбой **во время активации/удаления** — kernel выполняет это одной
  netlink-операцией. Либо переключение прошло целиком, либо не начиналось.

## Реакция на события

После первой успешной сборки агент остаётся в работе постоянно и ждёт три типа
событий.

### Событие 1. Сервер сообщил, что в БД что-то изменилось

Сервер сам присылает агенту сигнал в открытый стрим (никакого polling'а).
Шквал сигналов сворачивается в один по истечении окна
`extapi/svc/sgroups/sync-status/interval` (default `10s`) — см.
[«Защита от шквала событий»](#защита-от-шквала-событий).

Что делает агент дальше:

1. Скачивает полный снимок данных для этого узла.
2. Сравнивает со снимком прошлого цикла.
3. Если есть отличия — собирает и применяет новую nft-таблицу по алгоритму выше.
4. Если отличий нет — запоминает новую "версию" сервера и ждёт следующее уведомление.

### Событие 2. Изменилось состояние сети узла

Когда Linux добавляет / удаляет IP на любом интерфейсе или поднимает / опускает
линк, агенту прилетает netlink-уведомление. Действия:

1. Обновляет внутреннее представление сетевого состояния узла.
2. Запускает пересборку.
3. На шаге сравнения снимков:
   - Если изменился **локальный IP, относящийся к ресурсу `Host`** — снимок
     теперь отличается, ruleset перестраивается, новые IP попадают в set
     локального хоста, а на сервер уходит обновлённая информация о хосте.
   - Если изменение касалось интерфейса, который к локальному `Host` не имеет
     отношения (например, появилась/исчезла docker-bridge), — снимок будет
     эквивалентен прошлому, пересборка пропускается.

### Событие 3. Истекло время жизни DNS-записи

Для каждого FQDN, упомянутого в правилах, агент держит таймер: ровно к моменту
истечения TTL он переспросит у DNS адрес заново. Если адреса изменились —
**полная пересборка не запускается**. Вместо неё происходит точечный апдейт
одного nft-set: содержимое `NetIPv4-fqdn-<имя>` подменяется на новое
одной транзакцией.

Это самый дешёвый класс изменений — не трогаются ни цепочки, ни правила,
ни остальные set'ы. После апдейта планируется следующий резолв на новый TTL.

Если резолв упал (DNS не отвечает) — set остаётся таким, каким был. Следующий
плановый резолв повторит попытку. Если правил с этим FQDN больше нет (кто-то
удалил их со стороны сервера) — соответствующего set'а в текущей таблице
тоже нет, апдейт молча пропускается.

## Параллельность процессов

Внутри агента одновременно крутятся четыре независимых "контура":

| Контур | Что делает | Источник событий |
|---|---|---|
| Слежение за статусом сервера | держит gRPC-стрим, рассылает "версия сервера обновилась" с дебаунсом | gRPC `SyncStatus.Watch` |
| Слежение за сетью узла | держит netlink, рассылает изменения | netlink |
| Слежение за DNS-записями | по таймерам перерезолвит FQDN | время |
| Применение правил | принимает события от первых трёх и решает, что делать | внутренние очереди |

Все четыре поднимаются в момент запуска и работают до выхода из процесса. Они
обмениваются сообщениями через внутреннюю шину событий — ничего не пишут в
файлы, ничего не дёргают через сеть напрямую друг у друга.

## Восстановление после сбоев

Каждый из четырёх контуров устроен так, что в случае фатальной ошибки (упал
gRPC-стрим, ядро вернуло ошибку netlink, не получилось ни одной попыткой
применить транзакцию) — он завершает работу и сообщает остальным. Тогда:

1. Все четыре контура останавливаются.
2. Агент ждёт небольшую паузу (по умолчанию 10 секунд).
3. Все четыре контура поднимаются заново — с пустыми очередями, без
   "хвоста" предыдущих событий.
4. Цикл начинается с шага 1: подключение к серверу, поиск своего хоста,
   обход графа, сборка nft.

Этот подход избавляет от необходимости вручную восстанавливать промежуточные
состояния — после restart всё in-memory создаётся заново. Текущий ruleset в
ядре в этот момент не трогается: пока новый снимок не пришёл и не собран,
старая таблица продолжает обслуживать трафик.

## Полный жизненный цикл одной картинкой

```mermaid
sequenceDiagram
    autonumber
    participant OS as ОС / окружение
    participant A as sg-agent
    participant NL as netlink
    participant S as sg-server
    participant DNS as DNS
    participant K as kernel (nft)

    Note over A: Запуск процесса
    OS->>A: старт + чтение конфигурации
    A->>A: старт телеметрии
    A->>NL: подписка на изменения
    A->>S: gRPC connect
    A->>S: подписка на статус БД (Watch)
    A->>S: поиск ресурса Host по идентификатору
    S-->>A: Host (если найден)
    A->>S: публикация атрибутов и IP узла

    Note over A: Ожидание: первый netlink + первый статус
    NL-->>A: текущее состояние интерфейсов
    S-->>A: первый SyncStatus

    rect rgb(240,248,255)
        Note over A,K: Полный цикл сборки
        A->>S: скачивание снимка (Host → AG → Hosts / Networks / Services / Rules → удалённые AG/SVC)
        S-->>A: snapshot
        A->>DNS: резолв FQDN из правил
        DNS-->>A: A / AAAA + TTL
        alt снимок не изменился
            A-->>A: пропуск (лог: "did not change")
        else снимок изменился
            A->>K: создать спящую таблицу main-${ts}
            A->>K: добавить set'ы / chains / rules
            A->>K: активировать новую + удалить старые таблицы (атомарно)
            A->>S: публикация атрибутов узла
        end
    end

    loop Ожидание событий
        S-->>A: SyncStatus (БД изменилась)
        Note over A: → начать новый цикл сборки
        NL-->>A: AddrUpdate / LinkUpdate
        Note over A: → начать новый цикл сборки
        DNS-->>A: TTL истёк (по таймеру)
        A->>K: точечный апдейт fqdn-set'а (без пересборки)
    end

    Note over A: На любую фатальную ошибку — пауза ~10s, затем рестарт всех контуров
```

## Краткие гарантии

- Старый ruleset работает до тех пор, пока новый не собран и не активирован — окно
  без правил исключено.
- Полная пересборка запускается только при реальном изменении данных, относящихся
  к этому узлу. Лишние сигналы фильтруются.
- Изменения IP-адресов FQDN обновляются точечно, без пересборки.
- Любая ошибка приводит к перезапуску внутренних процессов агента, но не nft-таблицы:
  ядро видит правила всё это время.
- Сторонние nft-таблицы, созданные другими сервисами на узле, агент не удаляет.
