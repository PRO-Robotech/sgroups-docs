# База данных

`sg-server` хранит конфигурацию SGroups в PostgreSQL. БД здесь не пассивное
key-value-хранилище: в ней зафиксированы границы домена, идентичность
ресурсов, уникальность имен, ссылки между сущностями, правила инкрементальной
синхронизации и outbox для `List` / `Watch`.

Эта страница описывает схему как архитектурный контракт. По ней можно повторить
структуру БД: создать расширения и типы, развернуть таблицы, индексы, ограничения,
представления, функции синхронизации и триггеры событий в правильном порядке.

Основные таблицы приложения находятся в схеме **`sgroups`**. Таблица `sg_db_ver`
создается отдельно механизмом миграций и хранит состояние примененных миграций.

## Роль БД в архитектуре

PostgreSQL выполняет три задачи:

1. **Хранит ресурсную модель**: namespaces, address groups, hosts, networks,
   services, binding-таблицы и rule-таблицы.
2. **Поддерживает доменные инварианты**: уникальность имен в namespace,
   целостность ссылок, запрет изменения неизменяемых полей, отсутствие конфликтов
   transport-диапазонов у сервисов на одном AddressGroup.
3. **Обеспечивает инкрементальную синхронизацию**: глобальный `resourceVersion`,
   outbox событий, представления `vu_*` для API-проекции и функции `list_*` / `sync_*`.

Сервер не должен собирать эти гарантии вручную в памяти. Он работает поверх уже
нормализованной и проверенной схемы.

## Порядок развертывания

Схему нужно создавать в следующей последовательности:

1. Создать схему `sgroups`.
2. Подключить расширения PostgreSQL.
3. Создать enum-типы, домены, composite-типы, агрегаты и общие функции.
4. Создать служебные таблицы `tbl_global_resource_version`,
   `tbl_outbox_resource_events`, `tbl_sync_status`.
5. Создать таблицы ресурсов и binding-таблицы.
6. Создать базовые SQL-view `vu_*` для API-проекции.
7. Создать функции `list_*`, `sync_*`, функции outbox и триггеры.
8. Создать `tbl_rule_registry`.
9. Создать rule-таблицы.
10. Добавить индексы обратных ссылок на rule-таблицы.
11. Переопределить `vu_ag` и `vu_service`, чтобы они учитывали ссылки на правила.
12. Повесить триггеры `binding_rev`, `rule_rev`, immutable-field и outbox.

При нарушении порядка легко получить временно невалидное состояние: например,
правила зависят от `tbl_rule_registry`, outbox — от представлений,
а `vu_ag` и `vu_service` должны быть пересозданы после появления rule-таблиц.

## PostgreSQL-расширения

| Расширение | Зачем нужно | Когда создается |
| --- | --- | --- |
| `pgcrypto` | Генерация UUID через `gen_random_uuid()`. | На шаге инициализации схемы. |
| `hstore` | Хранение и фильтрация `labels`, `annotations`, selector-объектов. | На шаге инициализации схемы. |
| `btree_gist` | Поддержка индексируемых операций для диапазонов и проверок пересечений. | На шаге инициализации схемы. |
| `citext` | Case-insensitive FQDN для домена `sgroups.fqdn`. | Создается одновременно с FQDN-rule-таблицей, так как раньше домен `sgroups.fqdn` не нужен. |

## Доменные типы

### Enum-типы

| Тип | Значения | Назначение |
| --- | --- | --- |
| `sgroups.policy_action` | `DENY`, `ALLOW` | Действие политики или правила. |
| `sgroups.proto` | `tcp`, `udp`, `icmp` | L4-протокол. |
| `sgroups.traffic` | `both`, `ingress`, `egress` | Направление трафика. |
| `sgroups.ip_family` | `IPv4`, `IPv6` | IP-семейство правила. |
| `sgroups.resource_type` | См. ниже | Тип объекта в outbox, refs и API-представлениях. |

Значения `sgroups.resource_type`:

- **Ресурсы**: `Namespace`, `AddressGroup`, `Host`, `HostBinding`, `Service`, `ServiceBinding`, `Network`, `NetworkBinding`.
- **Rule-типы** (14 значений, по одному на каждую rule-таблицу):
  - AG-based: `Ag2AgRule`, `Ag2AgIcmpRule`, `Ag2IcmpRule`, `Ag2CidrRule`, `Ag2CidrIcmpRule`, `Ag2FqdnRule`, `Ag2SvcRule`, `Ag2SvcIcmpRule`.
  - Service-based: `Svc2SvcRule`, `Svc2FqdnRule`, `Svc2CidrRule`, `Svc2CidrIcmpRule`, `Svc2AgRule`, `Svc2AgIcmpRule`.


### Домены

| Домен | Базовый тип | Инвариант |
| --- | --- | --- |
| `sgroups.rname` | `text` | Имя ресурса: непустое, ограниченной длины, пригодное для API. |
| `sgroups.dname` | `text` | Отображаемое имя. |
| `sgroups.port_ranges` | `int4multirange` | Диапазоны портов в допустимом интервале. |
| `sgroups.icmp_types` | `int2[]` | ICMP-типы в диапазоне `0..255`. |
| `sgroups.fqdn` | `citext` | FQDN без учета регистра, с проверкой формата и длины. |

### Composite-типы

Схема активно использует composite-типы как контракт между SQL-функциями и сервером:

| Тип | Назначение | Где определен |
| --- | --- | --- |
| `sgroups.resource_ref` | Ссылка на другой ресурс: имя, namespace, тип ресурса. | Общая инициализация. |
| `sgroups.resource_id` | Человеческий идентификатор ресурса: name + namespace. | Общая инициализация. |
| `sgroups.res_selector` / `field_selector` | Фильтрация ресурсов в `list_*`. | Общая инициализация. |
| `sgroups.i4mr_any_intersect_state` | Состояние агрегата пересечения `int4multirange`. Лежит в основе всех проверок overlap портов. | Общая инициализация. |
| `sgroups.host_info` | Дополнительная информация хоста. | Создается вместе с `tbl_host`, поэтому формально не общий тип. |
| `sgroups.transport` / `transport_entry` | Описание портов/ICMP у Service. | Общая инициализация. |
| `sgroups.port_entries` / `icmp_entries` | Entries в rule-таблицах. | Создаются вместе с первой TCP/UDP- и первой ICMP-rule-таблицей. |
| `sgroups.endpoint` | Endpoint в API-проекции правил. | Создается одновременно с `tbl_rule_registry`. |
| `sgroups.row_of__rule` | Универсальный DTO для удаления правила через registry без знания конкретной rule-таблицы. | Создается одновременно с `tbl_rule_registry`. |
| `sgroups.row_of__*` (по одному на ресурс/rule-таблицу) | Входные DTO для `sync_*` функций. | Создается рядом с соответствующей таблицей. |

Эти типы важны: они делают SQL-функции устойчивым API-слоем между Go-кодом и
физическими таблицами БД.

## Служебный контур

### `tbl_global_resource_version`

Синглтон-таблица глобальной версии ресурсов.

| Поле | Тип | Ограничения | Назначение |
| --- | --- | --- | --- |
| `id` | `int2` | `pk`, значение `1` | Идентификатор singleton-строки. |
| `v` | `int8` | `not null`, `v >= 0` | Монотонно возрастающий глобальный `resourceVersion`. |

Каждое изменение ресурса получает новую глобальную версию. Эта версия используется
outbox-таблицей и `Watch/List-after-RV`.

### `tbl_outbox_resource_events`

Outbox событий ресурсов.

| Поле | Тип | Назначение |
| --- | --- | --- |
| `event_id` | `int8` | Локальный sequence-id события. |
| `ts` | `timestamptz` | Время записи события. |
| `resource_version` | `int8` | Глобальная версия события. |
| `resource_type` | `sgroups.resource_type` | Тип ресурса. |
| `event_type` | `text` | Тип события: добавление, изменение, удаление. |
| `object` | `jsonb` | API-представление ресурса без внутреннего `id`. |

Индексы:

| Индекс | Назначение |
| --- | --- |
| `outbox_resource_events_ts_idx` | Очистка устаревших событий по времени. |
| `outbox_resource_events_type_rv_event_id_idx` | Быстрое чтение событий по типу ресурса и `resourceVersion`. |

Outbox работает совместно с `NOTIFY`: после записи события БД отправляет уведомление
в канал вида `resource_<type>`, а потребители читают детали через функцию
списка outbox.

### `tbl_sync_status`

Служебная таблица состояния синхронизации.

| Поле | Назначение |
| --- | --- |
| `sync_count` | Счетчик синхронизаций, должен быть больше `0`. |
| `updated_at` | Время обновления, генерируется через SQL-функцию времени. |

## Общие правила ресурсных таблиц

Большинство ресурсных таблиц имеют одинаковый каркас:

| Поле | Назначение |
| --- | --- |
| `id` | Внутренний суррогатный PK, не должен попадать в API. |
| `uid` | Стабильный внешний идентификатор ресурса. |
| `name` | Имя ресурса внутри namespace. |
| `ns` | FK на `tbl_namespace.id`, если ресурс namespaced. |
| `display_name` | Человекочитаемое имя. |
| `labels` | Метки для фильтрации по селектору. |
| `annotations` | Произвольные аннотации. |
| `comment`, `description` | Текстовое описание. |
| `creation_timestamp` | Время создания. |
| `resource_version` | Текущая версия ресурса в API-проекции. |

Главный инвариант: **в API отдается `uid`, а не `id`**. Внутренний `id` нужен
только для FK и операций соединения (join).

## Ресурсные таблицы

### `tbl_namespace`

Корневой ресурс. Namespace задает область имен для остальных ресурсов и правил.

Ключевые ограничения:

| Ограничение | Смысл |
| --- | --- |
| `uid unique` | Стабильная глобальная идентичность namespace. |
| `name unique` | Namespace не имеет родительского namespace, поэтому имя глобально уникально. |
| GIN на `labels`, `annotations` | Быстрый поиск по селектору. |

### `tbl_ag`

AddressGroup — центральная сущность сетевой политики.

Дополнительные поля:

| Поле | Назначение |
| --- | --- |
| `default_action` | Действие по умолчанию (`DENY` / `ALLOW`). |
| `logs` | Включение логирования. |
| `trace` | Включение трассировки. |
| `binding_rev` | Ревизия состава bindings. |
| `rule_rev` | Ревизия набора правил, затрагивающих AG. |

Ограничения:

| Ограничение | Смысл |
| --- | --- |
| `(name, ns) unique` | Только один AddressGroup с данным именем в namespace. |
| `ns -> tbl_namespace(id)` | Удаление namespace каскадно удаляет AG. |
| GIN на `labels`, `annotations` | Фильтрация и поиск. |

`binding_rev` меняется при изменении `HostBinding`, `NetworkBinding`,
`ServiceBinding`. `rule_rev` меняется при изменении правил, где AG участвует как
локальный или удаленный endpoint.

### `tbl_network`

Описывает CIDR-сеть. Дополнительное поле `network` хранит CIDR, а `binding_rev`
показывает ревизию привязок сети к AG.

### `tbl_host`

Описывает хост. Дополнительные поля: `ips` для массива IP-адресов, `meta_info`
для JSONB-метаданных и `binding_rev` для ревизии привязок.

### `tbl_service`

Описывает сервис и его transport-профиль.

| Поле | Назначение |
| --- | --- |
| `transports` | Массив transport-описаний (`tcp`, `udp`, `icmp`). |
| `binding_rev` | Ревизия привязок сервиса к AG. |
| `rule_rev` | Ревизия правил, затрагивающих Service. |

Ключевой инвариант: transport-диапазоны внутри сервиса и между сервисами,
привязанными к одному AG, не должны конфликтовать. Это контролируется логикой `CHECK`,
агрегатами пересечений и триггерами.

## Binding-таблицы

Binding-таблицы связывают AddressGroup с конкретным ресурсом.

| Таблица | Связь | Уникальность |
| --- | --- | --- |
| `tbl_host_binding` | `tbl_ag` + `tbl_host` | `(ag, host)` и `(name, ns)` |
| `tbl_network_binding` | `tbl_ag` + `tbl_network` | `(ag, network)` и `(name, ns)` |
| `tbl_service_binding` | `tbl_ag` + `tbl_service` | `(ag, service)` и `(name, ns)` |

Общие поля binding-таблиц такие же, как у ресурсов: `id`, `uid`, `name`, `ns`,
`display_name`, `labels`, `annotations`, `comment`, `description`,
`creation_timestamp`, `resource_version`.

Важные инварианты:

1. Один и тот же ресурс нельзя дважды привязать к одному AG.
2. Binding имеет собственный `uid` и имя.
3. При изменении binding-таблиц обновляется `binding_rev` у затронутых AG и ресурса.
4. Для service binding дополнительно проверяется отсутствие пересечений transport-профилей
   сервисов на одном AG.

## Правила и `tbl_rule_registry`

Правила вынесены в отдельные таблицы по типам. Это осознанное решение: каждая
таблица хранит только поля, которые нужны конкретному виду правила, а общий
registry обеспечивает единую идентичность и уникальность имени.

### `tbl_rule_registry`

| Поле | Назначение |
| --- | --- |
| `uid` | PK правила. |
| `name` | Имя правила. |
| `ns` | Namespace правила. |

Ограничение `(name, ns) unique` в registry означает: **в одном namespace не может
быть двух правил с одинаковым именем, даже если они лежат в разных rule-таблицах**.

Каждая rule-таблица хранит свой `uid`, который является FK на
`tbl_rule_registry.uid` с каскадным удалением. Поэтому registry — это входная
точка удаления и cross-type uniqueness.

## Rule-таблицы

Слой правил разделен на таблицы по типу endpoints и L4-семантике. Это упрощает
ограничения: таблица с CIDR хранит `cidr`, таблица с FQDN — `fqdn`,
таблица с удаленными AG/Service — FK на удаленный ресурс, а ICMP-вариант не
обязан иметь `proto`.

У всех rule-таблиц есть общий ресурсный каркас:

| Поле | Тип | Назначение |
| --- | --- | --- |
| `id` | `int8` | Внутренний PK строки правила. |
| `uid` | `uuid` | Внешний идентификатор правила; FK на `tbl_rule_registry.uid`. |
| `name` | `text` | Имя правила. Дублируется с registry для локальной уникальности и удобства чтения. |
| `ns` | `int8` | Namespace правила. |
| `display_name` | `text` | Отображаемое имя. |
| `labels`, `annotations` | `hstore` | Метаданные и selector-атрибуты. |
| `comment`, `description` | `text` | Человекочитаемые пояснения. |
| `action` | `sgroups.policy_action` | `DENY` или `ALLOW`. |
| `traffic` | `sgroups.traffic` | Направление: `ingress`, `egress` или `both`. |
| `ip_v` | `sgroups.ip_family` | IP-семейство правила. |
| `creation_timestamp` | `timestamptz` | Время создания строки. |
| `resource_version` | `text` | Версия ресурса правила для API. |

Общие ограничения:

| Ограничение | Назначение |
| --- | --- |
| `uid unique` | Одно физическое правило на один внешний UUID. |
| `uid -> tbl_rule_registry(uid)` | Связывает физическую rule-таблицу с общим registry. |
| `(name, ns) unique` | Локальная уникальность имени внутри конкретной rule-таблицы. |
| `ns -> tbl_namespace(id)` | Правило принадлежит namespace. |
| GIN на `labels`, `annotations` | Поиск и фильтрация по селектору. |

Важно различать две уникальности:

1. `(name, ns)` в самой rule-таблице запрещает дубль внутри одного типа правил.
2. `(name, ns)` в `tbl_rule_registry` запрещает дубль **между всеми типами правил**.

### Семантика `local`, `remote`, `cidr`, `fqdn`

В rule-таблицах есть локальная и удаленная сторона.

| Поле | Где встречается | Смысл |
| --- | --- | --- |
| `aglocal` | AG-based-правила (AG → AG, AG → ICMP, AG → CIDR, AG → FQDN, AG → Service) | AddressGroup, к которому правило применяется локально. |
| `agremote` | AG → AG-правила, Service → AG-правила | Удаленный AddressGroup. |
| `svclocal` | Service-based-правила (SVC → SVC, SVC → FQDN, SVC → CIDR, SVC → AG) | Service, к которому правило применяется локально. |
| `svcremote` | Service → Service-правила, AG → Service-правила | Удаленный Service. |
| `cidr` | CIDR-правила | Удаленный CIDR-endpoint без отдельной ресурсной таблицы. |
| `fqdn` | FQDN-правила | Удаленный DNS-endpoint, без учета регистра, через домен `sgroups.fqdn`. |

В rule-таблицах со смешанными endpoints (`AG → Service` или `Service → AG`)
одновременно присутствуют поля `aglocal` или `svclocal` для локальной стороны
и `agremote` или `svcremote` для удаленной — оба ресурса участвуют в `refs`
и пересчете `rule_rev`.

Локальная сторона используется для пересчета `rule_rev` у AG или Service. Удаленная
сторона тоже попадает в `refs` и, если это AG или Service, участвует в обратном поиске
правил.

### `entries`

`entries` — это массив доменных записей, который задает L4-детали правила.

| Тип entries | Где используется | Что хранит |
| --- | --- | --- |
| `port_entries[]` | TCP/UDP-правила | Диапазоны портов и параметры transport-правила. |
| `icmp_entries[]` | ICMP-правила | ICMP-типы и коды. |

Для TCP/UDP-правил поле `proto` задает протокол. Для ICMP-таблиц протокол
подразумевается типом таблицы, поэтому отдельное поле `proto` не нужно.

## AddressGroup-based rule-таблицы

### `tbl_ag2ag_rule`

Правило между двумя AddressGroup для TCP/UDP-трафика.

| Поле | Назначение |
| --- | --- |
| `aglocal` | Локальный AG, для которого правило применяется. |
| `agremote` | Удаленный AG. |
| `proto` | TCP или UDP. В этой таблице обязателен. |
| `entries` | `port_entries[]`, портовые условия правила. |

Связи:

| FK | Поведение |
| --- | --- |
| `aglocal -> tbl_ag(id)` | Удаление AG удаляет правило. |
| `agremote -> tbl_ag(id)` | Удаление удаленной AG удаляет правило. |
| `ns -> tbl_namespace(id)` | Удаление namespace удаляет правило. |
| `uid -> tbl_rule_registry(uid)` | Удаление записи registry удаляет физическое правило. |

Индексы `aglocal` и `agremote` нужны для быстрого ответа на вопрос: какие правила
затрагивают конкретный AddressGroup.

### `tbl_ag2ag_icmp_rule`

ICMP-вариант правила между двумя AddressGroup.

| Поле | Назначение |
| --- | --- |
| `aglocal` | Локальный AG. |
| `agremote` | Удаленный AG. |
| `entries` | `icmp_entries[]`, ICMP-условия. |

`proto` здесь отсутствует: таблица уже означает ICMP. Это снижает риск записать
логически противоречивое правило вида «ICMP-правило с `proto=tcp`».

### `tbl_ag2icmp_rule`

Локальное ICMP-правило для AddressGroup без remote resource.

| Поле | Назначение |
| --- | --- |
| `aglocal` | AG, к которому относится правило. |
| `entries` | `icmp_entries[]`. |

Эта таблица нужна для случаев, когда правило описывает не связь «ресурс →
ресурс» или «ресурс → CIDR/FQDN», а ICMP-поведение локального AG.

Удаленный endpoint отсутствует, поэтому `refs` строятся только вокруг локального AG и
самого ресурса-правила.

### `tbl_ag2cidr_rule`

Правило AddressGroup → CIDR для TCP/UDP.

| Поле | Назначение |
| --- | --- |
| `aglocal` | Локальный AG. |
| `cidr` | Удаленная сеть в формате PostgreSQL `cidr`. |
| `proto` | TCP или UDP. |
| `entries` | `port_entries[]`. |

CIDR не является отдельным ресурсом SGroups, поэтому для удаленной стороны нет FK:
значение хранится непосредственно в строке правила.

### `tbl_ag2cidr_icmp_rule`

ICMP-вариант AddressGroup → CIDR.

| Поле | Назначение |
| --- | --- |
| `aglocal` | Локальный AG. |
| `cidr` | Удаленная сеть. |
| `entries` | `icmp_entries[]`. |

Как и у других ICMP-таблиц, `proto` не хранится отдельно.

### `tbl_ag2fqdn_rule`

Правило AddressGroup → FQDN.

| Поле | Назначение |
| --- | --- |
| `aglocal` | Локальный AG. |
| `fqdn` | Удаленное DNS-имя через домен `sgroups.fqdn`. |
| `proto` | TCP или UDP, если правило transport-based. |
| `entries` | `port_entries[]`. |

`fqdn` основан на `citext`, поэтому сравнение не зависит от регистра. Домен
также ограничивает формат и длину имени. Это важно: FQDN-endpoint хранится как
значение, а не как FK на отдельную таблицу ресурсов.

## Service-based rule-таблицы

Service-based-правила привязываются к `tbl_service`, а не к `tbl_ag`. Их назначение —
описывать политику на уровне сервисов и их transport-профилей.

### `tbl_svc2svc_rule`

Правило между двумя Service для TCP/UDP.

| Поле | Назначение |
| --- | --- |
| `svclocal` | Локальный Service. |
| `svcremote` | Удаленный Service. |
| `proto` | TCP или UDP. В этой таблице обязателен. |
| `entries` | `port_entries[]`. |

Индексы `svclocal` и `svcremote` нужны для `rule_rev` и `refs`: при изменении
правила оба сервиса должны знать, что набор затрагивающих правил изменился.

### `tbl_svc2fqdn_rule`

Правило Service → FQDN.

| Поле | Назначение |
| --- | --- |
| `svclocal` | Локальный Service. |
| `fqdn` | Удаленное DNS-имя через `sgroups.fqdn`. |
| `proto` | TCP или UDP, если правило transport-based. |
| `entries` | `port_entries[]`. |

У удаленной стороны нет FK, потому что FQDN — это значение, а не SGroups-ресурс.

### `tbl_svc2cidr_rule`

Правило Service → CIDR для TCP/UDP.

| Поле | Назначение |
| --- | --- |
| `svclocal` | Локальный Service. |
| `cidr` | Удаленная сеть. |
| `proto` | TCP или UDP. |
| `entries` | `port_entries[]`. |

Эта таблица используется для правил, в которых сервис обращается к внешней сети,
описанной CIDR, без создания отдельного Network-ресурса.

### `tbl_svc2cidr_icmp_rule`

ICMP-вариант Service → CIDR.

| Поле | Назначение |
| --- | --- |
| `svclocal` | Локальный Service. |
| `cidr` | Удаленная сеть. |
| `entries` | `icmp_entries[]`. |

`proto` отсутствует: ICMP задан типом таблицы.

## Смешанные AG↔Service rule-таблицы

Помимо «чистых» AG-based и Service-based правил, схема содержит четыре
rule-таблицы со смешанными endpoints: локальная сторона — AddressGroup, удаленная —
Service (или наоборот). Они нужны, чтобы выражать политики, в которых один
endpoint описан через состав хостов/сетей (AG), а другой — через transport-профиль
сервиса (Service).

### `tbl_ag2svc_rule`

Правило AddressGroup → Service для TCP/UDP.

| Поле | Назначение |
| --- | --- |
| `aglocal` | Локальный AG. |
| `svcremote` | Удаленный Service. |
| `proto` | TCP или UDP. В этой таблице обязателен. |
| `entries` | `port_entries[]`. |

### `tbl_ag2svc_icmp_rule`

ICMP-вариант AddressGroup → Service.

| Поле | Назначение |
| --- | --- |
| `aglocal` | Локальный AG. |
| `svcremote` | Удаленный Service. |
| `entries` | `icmp_entries[]`. |

### `tbl_svc2ag_rule`

Правило Service → AddressGroup для TCP/UDP.

| Поле | Назначение |
| --- | --- |
| `svclocal` | Локальный Service. |
| `agremote` | Удаленный AG. |
| `proto` | TCP или UDP. В этой таблице обязателен. |
| `entries` | `port_entries[]`. |

### `tbl_svc2ag_icmp_rule`

ICMP-вариант Service → AddressGroup.

| Поле | Назначение |
| --- | --- |
| `svclocal` | Локальный Service. |
| `agremote` | Удаленный AG. |
| `entries` | `icmp_entries[]`. |

Для всех четырех таблиц действует общий принцип FK:

| FK | Поведение |
| --- | --- |
| `aglocal -> tbl_ag(id)` или `svclocal -> tbl_service(id)` | Удаление локального ресурса удаляет правило. |
| `agremote -> tbl_ag(id)` или `svcremote -> tbl_service(id)` | Удаление удаленного ресурса удаляет правило. |
| `ns -> tbl_namespace(id)` | Удаление namespace удаляет правило. |
| `uid -> tbl_rule_registry(uid)` | Удаление записи registry удаляет физическое правило. |

При изменении смешанного правила обновляются `rule_rev` **обоих** ресурсов: и AG, и Service.

## Как rule-таблицы влияют на `refs` и ревизии

После создания rule-таблиц миграция `resource_refs` добавляет функции обратного
поиска правил и пересоздает представления `vu_ag` / `vu_service`.

Итоговая модель:

| Ресурс | Что попадает в `refs` |
| --- | --- |
| `AddressGroup` | Bindings на host/network/service и все AG-based-правила, где AG — локальный или удаленный. |
| `Service` | Service bindings и все Service-based-правила, где сервис — локальный или удаленный. |

`rule_rev` обновляется триггерами при изменении rule-таблиц. Это позволяет серверу
и подписчикам отличать изменение самого ресурса от изменения набора правил,
которые на него ссылаются.

## Внешние ключи и удаление

| Связь | Поведение |
| --- | --- |
| Ресурс → namespace | `ON DELETE CASCADE`, `ON UPDATE RESTRICT`. |
| Binding → AG/resource | `ON DELETE CASCADE`, `ON UPDATE RESTRICT`. |
| Rule → namespace | `ON DELETE CASCADE`, `ON UPDATE RESTRICT`. |
| Rule → registry | `ON DELETE CASCADE`. |
| Rule → local/remote AG или Service | `ON DELETE CASCADE`, `ON UPDATE RESTRICT`. |

Удаление namespace удаляет все ресурсы и правила внутри него,
а удаление AG/Service — связанные bindings и правила, в которых они участвуют.

## Индексы

| Группа | Индексы | Назначение |
| --- | --- | --- |
| Уникальность имени | `(name, ns)` | Быстрое разрешение ресурса по API-идентификатору. |
| Уникальность UID | `uid unique` | Быстрая синхронизация по стабильному идентификатору. |
| Поиск по селектору | GIN на `labels`, `annotations` | Фильтрация ресурсов. |
| Уникальность binding | `(ag, host)`, `(ag, network)`, `(ag, service)` | Запрет дубликатов binding-таблиц. |
| Обратный поиск правил | `aglocal`, `agremote`, `svclocal`, `svcremote` | Быстрый поиск правил, затрагивающих AG/Service. |
| Курсор outbox | `(resource_type, resource_version, event_id)` | Инкрементальное чтение событий. |

## Представления `vu_*`

Физические таблицы — это не внешний API. Для чтения сервер использует представления:

| Представление | Назначение |
| --- | --- |
| `vu_namespace` | Namespace в API-представлении. |
| `vu_ag` | AddressGroup + namespace + `refs` на bindings и правила. |
| `vu_network` | Network + namespace + `refs` на bindings. |
| `vu_host` | Host + namespace + `refs` на bindings. |
| `vu_service` | Service + namespace + `refs` на bindings и правила. |
| `vu_*_binding` | Binding с развернутыми ссылками на AG/ресурс. |
| `vu_*_rule` | Rule в API-представлении с endpoint-структурами. |

Представления выполняют важную нормализацию:

- заменяют внутренние `id` на имена и namespaces;
- строят `refs`;
- собирают composite-структуры;
- формируют JSON-объекты для outbox;
- скрывают технические колонки, которые не должны попадать во внешний контракт.

После добавления rule-таблиц представления `vu_ag` и `vu_service` переопределяются:
в их `refs` попадают не только bindings, но и правила, которые затрагивают ресурс.

## Функции `list_*`

Функции `list_*` — это слой чтения для сервера.

Ресурсные `list_*` читают из `vu_*` и применяют фильтрацию по селектору.
Binding- и rule-функции читают соответствующие представления без сложной логики селекторов.

Главная идея: клиент сервера не должен знать физические таблицы. Он запрашивает
готовую API-проекцию.

### Fast-path для простых селекторов

`list_services`, `list_ag` и `list_hosts` имеют три ветви исполнения, чтобы
не материализовать `vu_*.refs` без необходимости:

1. **Match-all**: если в `selectors[]` есть хотя бы один селектор без name, namespace,
   refs и labels — возвращается весь `vu_*` без фильтрации.
2. **Complex selectors**: если хотя бы один селектор содержит непустой `refs[]`
   или хотя бы один label-ключ — используется общая логика `match_res_selectors`.
3. **Simple selectors** (fast-path): если все селекторы фильтруют только по
   `name` и/или `namespace`, проверка делается через `EXISTS (... unnest(selectors) ...)`
   без вызова `match_res_selectors` и без обращения к `refs`.

Чтобы fast-path работал, добавлены индексы `host_binding_host_idx`
на `tbl_host_binding(host)` и `service_binding_service_idx` на `tbl_service_binding(service)` —
они ускоряют построение `vu_*.refs` для веток complex/match-all, когда `refs`
все-таки нужны.

## Функции `sync_*`

Функции `sync_*` — это слой записи. Они принимают строки composite-типа `row_of__*`
и операцию `sync_op`.

| Операция | Поведение |
| --- | --- |
| `ups` | Создать или обновить ресурс по `uid` либо по `(name, ns)`. |
| `del` | Удалить ресурс по `uid` либо по `(name, ns)`. |

Для каждой ресурсной таблицы и для каждой rule-таблицы существует своя `sync_*`-функция:
`sync_namespaces`, `sync_address_groups`, `sync_networks`, `sync_hosts`, `sync_services`,
`sync_host_bindings`, `sync_network_bindings`, `sync_service_bindings`,
а также `sync_ag2ag_rule`, `sync_ag2ag_icmp_rule`, `sync_ag2icmp_rule`, `sync_ag2cidr_rule`,
`sync_ag2cidr_icmp_rule`, `sync_ag2fqdn_rule`, `sync_ag2svc_rule`, `sync_ag2svc_icmp_rule`,
`sync_svc2svc_rule`, `sync_svc2fqdn_rule`, `sync_svc2cidr_rule`, `sync_svc2cidr_icmp_rule`,
`sync_svc2ag_rule`, `sync_svc2ag_icmp_rule`.

### Удаление правил через registry

Удаление правила выполняется отдельной функцией **`sync_rule_via_registry(op, row_of__rule)`**:
она принимает только `uid`/`name`/`namespace` и сама находит, в какой именно
rule-таблице лежит запись (через `tbl_rule_registry`). Это позволяет удалить
логическое правило, не зная его типа.

Для хоста дополнительно существуют `sync_host_info` и `sync_host_ipset` —
они синхронизируют служебные части ресурса Host (meta-info и IP-набор) без
полной перезаписи строки.

## Outbox и Watch

Outbox строится вокруг глобального `resourceVersion`.

Типичный путь изменения:

1. Строка ресурса вставляется, обновляется или удаляется.
2. Триггер `BEFORE` увеличивает `tbl_global_resource_version.v`.
3. Триггер `AFTER` или `BEFORE` формирует событие outbox.
4. Payload берется из соответствующего `vu_*`.
5. Внутренний `id` удаляется из JSON.
6. БД отправляет `NOTIFY` по каналу ресурса.
7. Подписчик Watch дочитывает события из `tbl_outbox_resource_events`.

Outbox хранит события ограниченное время. Если клиент приходит со слишком старым
`resourceVersion`, функция чтения outbox должна вернуть сигнал, что курсор истек,
и клиент обязан выполнить полный `List`.

## Триггеры

В таблице ниже перечислены **функции триггеров** (PL/pgSQL): на каждую функцию навешено
множество конкретных триггеров с именами вида `trg_<entity>_<purpose>`.

| Функция триггера | Где применяется | Назначение |
| --- | --- | --- |
| `inc_resource_version_cnt` | Ресурсы, bindings, rules | Выдает новую глобальную версию. |
| `capture_outbox_resource_events_trg` | Ресурсы, bindings, rules | Пишет событие в outbox. |
| `notify_outbox_resource_events_trg` | Outbox | Отправляет PostgreSQL `NOTIFY`. |
| `forbid_fields_update_trg` | Ресурсы, bindings, rules | Запрещает изменение неизменяемых полей (`uid`, `name`). |
| `upd_refs_trg` | Binding-таблицы и rule-таблицы | Обновляет `binding_rev` и `rule_rev` у затронутых ресурсов (единая функция для всех типов ссылок). |
| `check_transport_overlap_on_service_trg` | `tbl_service` | Запрещает конфликтующие транспорты внутри сервиса и между сервисами на одном AG при апдейте `tbl_service`. |
| `check_transport_overlap_on_binding_trg` | `tbl_service_binding` | Та же проверка при привязке сервиса к AG через `tbl_service_binding`. |
| `check_rule_port_entries_no_overlap_trg` | TCP/UDP rule-таблицы | Запрещает пересекающиеся диапазоны портов в `entries` одного правила. |
| `check_rule_icmp_entries_no_overlap_trg` | ICMP rule-таблицы | Запрещает пересекающиеся ICMP-типы в `entries` одного правила. |

## Инварианты схемы

1. **`uid` — внешний идентификатор, `id` — внутренний FK.**
   В API нельзя отдавать `id`.
2. **`resourceVersion` глобален для всей схемы.**
   Его нельзя делать per-table, иначе `Watch` потеряет общий порядок событий.
3. **Namespace владеет ресурсами.**
   Удаление namespace каскадно удаляет зависимые строки.
4. **Имя ресурса уникально в namespace.**
   Для namespace имя уникально глобально.
5. **Имя правила уникально между всеми типами правил в одном namespace.**
   Это обеспечивает `tbl_rule_registry`.
6. **Bindings не дублируют пары.**
   Один host/network/service нельзя дважды привязать к одному AG.
7. **Транспорты сервисов на одном AG не конфликтуют.**
   Конфликт портов или ICMP внутри сервиса и между сервисами одного AG блокируется
   двумя триггерами: на изменение `tbl_service` и на изменение `tbl_service_binding`.
8. **Внутри одного правила `entries` не пересекаются.**
   Нельзя в одной строке rule-таблицы указать два пересекающихся диапазона портов
   или один и тот же ICMP-тип дважды. Проверяют `check_rule_port_entries_no_overlap_trg`
   и `check_rule_icmp_entries_no_overlap_trg`.
9. **`refs` должны отражать не только bindings, но и правила.**
   Поэтому `vu_ag` и `vu_service` зависят от слоя правил.
10. **Payload outbox строится из представления, а не из исходной таблицы.**
    Это сохраняет внешний контракт API.
11. **Неизменяемые поля нельзя менять через `UPDATE`.**
    Для переименования или смены идентичности нужен отдельный контролируемый сценарий.

## Минимальный чек-лист восстановления схемы

Чтобы повторить эту БД с нуля, нужно реализовать:

1. `CREATE SCHEMA sgroups`.
2. `CREATE EXTENSION pgcrypto`, `hstore`, `btree_gist` (`citext` создается позже, вместе с FQDN-rule-таблицей).
3. Все enum-типы: `policy_action`, `proto`, `traffic`, `ip_family`, `resource_type`.
4. Домены и composite-типы: ссылки на ресурсы, селекторы, транспорты, entries, `i4mr_any_intersect_state`, `row_of__*`.
5. Служебные таблицы: версии, outbox, статус синхронизации.
6. Таблицы ресурсов: namespace, AG, network, host, service.
7. Binding-таблицы.
8. Rule registry, `endpoint`, `row_of__rule`.
9. Все rule-таблицы (включая четыре смешанные `ag2svc*` / `svc2ag*`). Расширение `citext` создается на шаге FQDN-rule-таблицы.
10. Внешние ключи, уникальные ограничения, `CHECK` и индексы GIN/B-tree.
11. Представления `vu_*` (после rule-таблиц — переопределить `vu_ag` и `vu_service`, чтобы они учитывали ссылки на правила).
12. Функции `list_*` (с fast-path для простых селекторов в `list_services`, `list_ag`, `list_hosts`).
13. Функции `sync_*`, включая `sync_rule_via_registry`.
14. Функции и триггеры outbox, а также `NOTIFY`.
15. Триггеры на неизменяемые поля.
16. Триггеры `upd_refs_trg` для bindings и rules (единая функция).
17. Триггеры `check_transport_overlap_on_service_trg` и `check_transport_overlap_on_binding_trg`.
18. Триггеры `check_rule_port_entries_no_overlap_trg` и `check_rule_icmp_entries_no_overlap_trg` на rule-таблицах.
19. Дополнительные индексы из миграции оптимизации: `host_binding_host_idx`, `service_binding_service_idx`.

После этого схема будет не просто набором таблиц, а полной серверной моделью
SGroups с API-проекцией, инкрементальной синхронизацией и гарантиями целостности.
